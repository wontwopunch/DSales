## 0. ⚠️ 착수 전 반드시 읽을 것 — 확정 / 미확정 경계

이 문서는 **화면 기획서(29p) + 7/8 미팅록 + 자체 분석**으로 만들었습니다. **소스·DB를 열어본 결과가 아닙니다.** 아래는 근거 강도가 다르므로 구분해서 다루십시오.

### 0.1 ✅ 확정 — 근거 있음

| 항목 | 근거 |
|---|---|
| **주문 유입 경로 2갈래** | 아래 0.2 |
| DSRE = **더프리미엄모의고사 운영 웹시스템** | 견적서 2번 "더프리미엄모의고사(DSRE) 운영 사이트" + 2-1~2-4 전부 모의고사 운영 기능(전년도 정보/실시일자/성적표 부수/예약발송). ⚠️ 초기에 "책 배송 물류회사"로 안내받았으나 **오정보** |
| DS물류 **시스템 연동 없음** | 발주처 요청. 수신 알람만 |
| 재고·매출·채권 = **매출프로그램 자체 완결** | P.7 창고 컬럼 / P.8 이중 장부 / P.13 미결 정산 |
| 세무 종착지 = **홈택스 엑셀** (더존 아님) | P.22 "홈택스 계산서 신고 업로드 파일 양식" |
| **(B) 동일 재현** (레거시 공존 없음) | 발주처 확정 |

### 0.2 ★ 주문 유입 경로 — 2갈래 (v0.3 확정)

```
[특약점] order.dsdo.co.kr
   ├─ 더프리미엄모의고사(전과목)  → DSRE → 매출프로그램  ※매출일괄등록이 read
   └─ 교재 · 과목별 · 기타 전부   → 매출프로그램 직결     ※주문/출고관리로 유입
```

**⚠️ v0.2의 "프론트가 매출프로그램에 직접 붙는 게 아니라 DSRE를 경유한다"는 서술은 과잉 일반화였습니다. 더프모에만 해당합니다.**

**근거**

| 경로 | 근거 |
|---|---|
| **더프모 → DSRE** | 기획서 P.14 "DSRE에 입력된 **더프** 출고내역을 매출로 일괄 등록" / 견적서 1-5 "order 주문 시 담당자정보 자동 불러오기 · **DSRE 접수완료 내역** 자동 표시" / 2.2 괴리(100 vs 50) |
| **교재·기타 → 직결** | 기획서 P.1 "거래명세서는 **매출프로그램 작업요청서 화면 연동**" / P.5 "**간편 신청 사이트** 주문 확인", 접수경로=온라인/수기 / DSRE는 기획서 전체에서 **더프 맥락에서만** 등장. 교재 맥락엔 한 번도 없음 / P.26·28 물류 등록처는 리브커넥스·학력개발본사·엔컴·디랩·대성인쇄 — **DSRE 없음** |

**이것이 개발에 주는 영향**

1. **접수경로가 3개다.** 기획서 P.5 필터는 `전체/온라인/수기`지만 실제 유입은 **order 직결 / DSRE 경유 / 수기**. → 온라인을 둘로 나눌지 결정 필요(🟡)
2. **상태 동기화 경로도 2개다.** 교재는 `매출프로그램 → order` 직결, 더프모는 `매출프로그램 → DSRE → order` 2단. **같은 "상품준비중"을 두 경로로 밀어야 함**
3. **신규 [과목별 모의고사/교재] 메뉴는 DSRE와 무관하다.** DSRE=**더프리미엄**(전과목), 신규 메뉴 상품=**과목별**(CONSTANT 수학, D.ARCHIVE 국어/과탐 등). **다른 상품군** → **매출프로그램 직결.** 이번 프로젝트의 본체가 여기

### 0.3 🔴 미확정 — 소스로 확인할 것 (착수 즉시)

| # | 항목 | 왜 지금인가 |
|---|---|---|
| 1 | **레거시 스택이 정말 ASP인가** | 아래 0.4. **견적서 전제조건 1번이 "ASP 소스 제공"** — 틀리면 견적서 신뢰도 문제 |
| 2 | **D.LINK 주문의 저장 경로** | 기획서 P.2(신규)와 P.3(기존 모의고사)에 **동일 주문**이 있음(2026-06-26 / D.LINK 시즌1 SET / PK 대치스파르타 배곧학원 / 20). 목업 샘플 재활용일 수도 있으나, **D.LINK가 지금 기존 [모의고사] 메뉴로 팔린다면 DSRE로 가고 있다는 뜻** → 신규 메뉴로 옮기면 **경로가 바뀜** |
| 3 | order ↔ 매출프로그램 **DB 공유 여부** | 공유면 백오피스 스키마 재설계가 **프론트를 연쇄로 깨뜨림**. 계약엔 프론트가 "기능 추가"로만 잡혀 있음 → **재견적 사유** |
| 4 | `DB 변경` 메뉴 정체 | 연도별·법인별 DB 분리면 **마이그레이션 범위·데이터 모델이 통째로 바뀜.** "논외"로 넘길 수 없음 |
| 5 | 비즈니스 로직 소재(프로시저 vs 클라이언트) | 2-tier면 프로시저에 몰려 있을 확률 높음. **이관 공수 최대 변수** |

### 0.4 ⚠️ "기존 ASP" 표기 보류

**기획서 스크린샷은 ASP일 수 없습니다.**

| 근거 | 판정 |
|---|---|
| P.21·23·24·28 `2026년 5월 1일 금요일 ▦▾` | **Win32 DateTimePicker**(한국어 긴 날짜). 브라우저 렌더 불가 |
| P.4 `창 > 계단식 배열 / 바둑판식 배열` + `1통합조회 2교재출고` | **Win32 MDI.** 웹에 없는 개념 |
| P.18 "현재 설정으로는 2022년까지만 조회 가능합니다" 모달 | Win32 MessageBox |
| P.14·17 그리드 + `Excel 저장` | 네이티브 위젯 |

→ **VB6 / Delphi / .NET WinForms 계열 데스크톱 앱**으로 추정. 반면 `order.dsdo.co.kr/user/open.**do**` → **`.do`는 Java(Struts/Spring) 규약** — 특약점 사이트는 이미 Java.

**본 문서의 "ASP" 표기는 전부 검증 대기 상태입니다.** 확인 즉시 일괄 정정하십시오. `ASP → Java`와 `VB6 → Java`는 다른 문제입니다(전자는 웹→웹, 후자는 **2-tier 데스크톱 → 3-tier 웹 아키텍처 이전**).

### 0.5 ⚠️ 프론트 화면 성격 — P.1~2 vs P.3

| 페이지 | 제목 | 성격 |
|---|---|---|
| **P.1~2** | "[프론트] 간편 신청 사이트 **신규 메뉴 생성 건**" | **희망 목업.** 실제 화면 아님 |
| **P.3** | "[프론트] 간편 신청 사이트 **기존 메뉴 수정 건**" | **실제 운영 화면 + 버그 리포트** |

P.3 원문: *"[상품준비] 상태는 **현재 조회 버튼만 노출되고 있고**, 취소 버튼은 비활성화 처리**되어있으나**, **실제로** 조회 버튼 클릭 시 하단에 수정이 가능한 상태"* → 실제 화면의 결함.

P.1 주석도 *"조회/수정은 **현재 모의고사 조회/수정 메뉴 참고**, 재고현황은 **현재 [교재] 메뉴의 재고현황 메뉴 참고**"* → 현재 화면 존재.

**의미**: 사이트에 모의고사 신청이 이미 돌아가고 있고 **소스를 확보했으므로 신규 메뉴는 백지 설계가 아니라 기존 모듈 복제·변형**이 가능. **공수가 다름.**
**주의**: P.3 버그가 **서버 검증 누락**이면 지금도 API 직접 호출로 뚫림 → **보안 이슈로 격상**

## 1. 시스템 개요

### 1.1 도메인 한 문장
> **"위탁(선출고 후정산) 비즈니스를 지탱하는 재고·매출 상태머신."**
> 대성 학력개발은 교재·모의고사(시험지)를 제작해 특약점·학원·학교에 공급한다. 매출프로그램은 주문 접수 → 재고 → 출고 → 매출 확정 → 수금 → 세무신고 → 물류작업 전 과정을 관리하는 백오피스다.

### 1.2 판매 방식 2종(복잡도의 근원)
- **정상 판매**: 출고 즉시 매출 확정.
- **위탁 판매(선출고 후정산)**: 먼저 보내되 아직 매출 아님. 거래처가 정산해줄 때 매출 확정. → **재고 상태와 매출 상태를 각각 별도 추적**해야 함(이중 장부).

### 1.3 시스템 지형 (v0.3)
| 시스템 | 성격 | 우리 관여 |
|---|---|---|
| 더존 출판 ERP | 상위 회계/결산 | 실시간 연동 거의 없음(있어도 파일 수준) — 재무팀 확인 |
| **매출프로그램(백오피스)** | **전면 재구축 대상**. 레거시 스택 🔴미확정(0.4) | **본 문서 전부** |
| **DSRE** | ✅ **더프리미엄모의고사 운영 웹시스템**(0.1). 더프모 주문의 실질 처리 | ① 매출프로그램과 **데이터 연동**(read) ② **사이트 자체 수정 4건**(8장) |
| **order.dsdo.co.kr**(특약점) | 특약점 주문 사이트(**운영 중**). Java(`.do`) 추정 | ① **신규 메뉴 4화면**(P.1~2 목업 기반) ② **기존 버그 수정**(P.3) ③ **유지보수(계약 조건 미정 🔴)** |

### 1.4 일정
- **개발 목표: 2026-10-31** / **실사용 개시: 2026-11-말**. 인력 집중 투입 필요(Phase 우선순위 결정 요인).

## 2. 시스템 구성 & 대상 범위

### 2.1 데이터 흐름 (v0.3 — 2갈래)
```
[특약점]
   └─ order.dsdo.co.kr (간편신청 사이트, 운영 중)
          │
          ├──── 더프리미엄모의고사(전과목) ────┐
          │                                    ▼
          │                        [ DSRE ] (더프모 운영 웹시스템)
          │                                    │ 더프 출고/응시 데이터
          │                                    ▼ (매출일괄등록이 read → 현재는 수기라 100 vs 50 괴리)
          │                                    │
          └──── 교재 · 과목별 · 기타 전부 ─────┤ (매출프로그램 직결 — 주문/출고관리 유입)
                                               ▼
   ┌──────────────────────────────────────────────┐
   │            매출프로그램 (백오피스) — 재구축 대상       │
   │  주문/출고 · 매출 · 마감 · 물류/작업 · 기초관리       │
   │  자체 재고 엔진 + 이중 장부 + 위탁 미결 정산 + 채권/여신 │
   └──────────────────────────────────────────────┘
          │                         │
          ▼(엑셀)                    ▼(문서/데이터)
      홈택스(세금계산서)        물류처(리브커넥스/본사) · DS물류(수신 알람만, 연동X)
          ·                         ·
      더존(회계결산 잔여 여부 — 재무팀 확인, 있어도 파일 수준)
```

### 2.2 현재 핵심 니즈(미팅 확인)
- **DSRE ↔ 매출프로그램 매출 괴리 자동화**: 현재 DSRE는 100부인데 매출프로그램엔 실제 매출 50부만 수기 입력 → **DSRE에 안 들어가도 매출프로그램에서 매출수불 확인 가능**하도록 자동화하는 것이 핵심.
- **매출일괄등록**: DSRE에서 그대로 가져오는 방식.
- **매출액 정리**: 현재 수기 → 자동 표시 개선.
- **대시보드 신규**: 현재 RAW 엑셀 받아 수기 그래프 → 프로그램 내 대시보드로.

### 2.3 재구축 방침
- **(B) 동일 재현 확정**: 레거시 존치가 아니라, **기능/로직을 Java/React로 동일 재현**(레거시 공존 없음).
- 화면 유형: **[신규] 완전 신규 설계 / [개편] 기존+새 개념 반영 / [🔁재현] 기존 로직 이식**.
- ⚠️ **"재현"이 "무공수"가 아니다.** 웹 전환이므로 UI는 전부 신규. 재현되는 것은 **쿼리·계산식·집계 규칙**뿐. 게다가 이번 개편으로 신규 컬럼(`창고ID`·`출고유형6`·`미결여부`·`원본위탁출고번호`·`BOM관계`·`정산수량`)이 대거 추가되므로 **스키마 위에 얹힌 모든 화면이 영향받음**.
- **재현 강도 3등급**(견적·일정에 반영):

| 등급 | 대상 | 성격 |
|---|---|---|
| **A. 신규 설계** | 재고 엔진, 위탁 미결 정산, 주문 통합등록, 매출 등록, 대시보드 2종, 월마감 | 새로 짬 |
| **B. 최소 재현** | 마감관리 5, 응시현황 2, 물류작업비, 폐기, 수불부 | **조회 전용 = 쿼리 이식 + 그리드 바인딩.** 저비용 |

## 4. 핵심 데이터 모델 (5대 축 + α)

> 재구축 설계의 뼈대. 이 축들이 서로 얽혀 복잡도가 발생한다. (컬럼은 예시, 실제 스키마는 기존 DB 확보 후 확정)

### 4.1 상품(Product)
- **단품 ↔ 세트(BOM)**: 재고는 단품(회차별 낱권) 단위. 세트 재고는 단품을 **조립(BOM)** 해야 생성.
- **자체 교재 ↔ 외부 콘텐츠**: 자체는 매입원가 없음. 외부(이감·한석원 등)는 **매입 발생** → 순매출 계산식 다름.
```sql
products ( id, code, name, content_type,  -- SELF(자체) | EXTERNAL(외부매입)
           is_set BOOLEAN, ... )
bom_items ( parent_product_id, child_product_id, ratio )  -- 완제품→구성품+비율
```

### 4.2 창고(Warehouse) / 재고(Inventory)
- **물류창고(실물) ↔ 위탁창고(가상)**. 재고는 **상품 × 창고** 단위 원장.
```sql
warehouses ( id, name, type )   -- PHYSICAL(물류) | CONSIGN(위탁/가상)
inventory ( product_id, warehouse_id, qty )
inventory_txn ( id, product_id, warehouse_id, txn_type, qty, ref_no, created_at )
-- txn_type: INBOUND | OUTBOUND | TRANSFER | BOM_ASSEMBLE | BOM_DISASSEMBLE | DISPOSE | RETURN
```

### 4.3 장부(Ledger) — ★이중 장부(가장 중요)
- **물류(재고) 장부**: 물건 위치·수량. 나가면 즉시 차감.
- **재무(매출) 장부**: 매출 금액. 위탁은 나가도 즉시 매출 아님(미결).
- **핵심 규칙**: 위탁출고 → 물류 장부 즉시 차감 / 재무 장부 **미결(보류)** → 나중 정산 시 재무로 전환.

### 4.4 매출유형 / 회계구분 (룩업 자동매핑)
| 출고유형(6) | 회계구분(3) | 재고위치 |
|---|---|---|
| 정상출고 | 매출 | 물류창고(실물) |
| 위탁출고 | 매출(미결→확정) | **위탁창고(가상)** |
| 증정용 | 무상 | 물류창고 |
| 교사용 | 무상 | 물류창고 |
| 반품 | 반품 | 물류창고(음수) |
| 취소 | 반품 | — |
```sql
sales_txn ( id, product_id, partner_id, sales_type,  -- NORMAL(일반) | CONSIGN(위탁)
            out_type, acct_type, qty, supply_amount, tax, total,
            source_out_no,  -- 위탁 확정 시 원본 출고번호 역추적
            ... )
```

### 4.5 매입(Purchase) — 외부 콘텐츠
```sql
purchases ( id, product_id, partner_id, qty, purchase_return_qty, amount, ... )
-- 순매출: 매출(A) − 매입(B) = 이익, 이익률
```

### 4.6 채권(AR)/여신·담보
```sql
ar_ledger ( partner_id, opening_balance, ... )  -- 미수금 = 매출 − 수금 (러닝밸런스)
receipts ( id, partner_id, receipt_type, pay_type, amount, note_no, due_date, bank, ... ) -- 수금/어음
collaterals ( partner_id, amount, rate, expires_at, kind, content )  -- 담보/여신
```

### 4.7 목표(예산) — 대시보드용(실적과 독립)
```sql
sales_targets ( year, month, scope, scope_ref_id, target_amount )  -- 전사/제품별/거래처별
```

### 4.8 권한(RBAC) — 횡단
- 같은 화면·기능도 **부서별(사업부/재무팀/물류팀)** 접근 상이. 예: 입고/대체 화면에서 물류팀은 "일반입고"만.

### 4.9 🆕 매출 월마감(컨펌/잠금)
- 한 달 매출 처리를 모두 끝낸 뒤 **[월마감 컨펌] 버튼**을 누르면 해당 월 매출 데이터가 **수정 잠금**된다(더 이상 등록/수정/취소 불가).
```sql
period_locks (
  id, year, month, branch_id NULL,   -- (지점/계열사 구분 필요 시)
  status,        -- OPEN | CONFIRMED(잠금)
  confirmed_by, confirmed_at,
  reopened_by, reopened_at            -- 잠금 해제(재오픈) 이력 — 권한자만
)
```
- 잠금 판정: 매출/수금/전표 등 **쓰기 API는 대상 연월의 `period_locks.status`를 먼저 검사** → `CONFIRMED`면 거부(에러 `PERIOD_LOCKED`).
- **재오픈(잠금 해제)** 은 상위 권한(예: 재무팀/관리자)만 가능하며 이력 남김.

## 5. 3대 핵심 로직 (프로젝트의 심장)

> 난이도의 대부분이 여기 집중. 화면보다 **백엔드 도메인 정합성**이 핵심.

### 5.1 [로직 A] 재고 엔진 — 입고 / 이고 / BOM (난이도: 상)
모든 작업은 **원자적 트랜잭션**(중간 실패 시 전체 롤백) + 음수재고 방지.
- **(1) 일반 입고**: 인쇄소(예: 대성인쇄) → 물류창고 최초 입고. 단품 단위, **입고단가(원가) 기록**. `물류창고.단품 += 입고수량`.
- **(2) 단순 이고**: 매출 없이 위치 이동(물류창고→위탁창고 등). `출발 −=, 도착 +=`. **★위탁출고 선택 시 이 이고가 자동 백그라운드 생성**(주문 화면과 한 트랜잭션).
- **(3) BOM 조립/해체**: 세트 조립 100개 시 — 완제품 +100, 구성품 각 −(BOM비율×100). 해체는 부호 반대. **조립/해체에 물류작업비(포장비 등) 발생** → 물류비 연동.
```
[완제품] 국어세트 +100
 └ 국어1회차 −100 · 국어2회차 −100 · 국어3회차 −100   (BOM비율 1:1:1)
```
- **(4) 검산 = 제품수불부**: `이월 + 입고 − 매출 − 교사용 − 반품 ± 폐기 ± 대체 = 현재재고`. 이 표가 재고 엔진의 **인수 테스트 명세**.

### 5.2 [로직 B] 위탁 미결 정산 — 부분정산 누적 (난이도: 상)
```
1) 위탁출고 → 물류 장부 차감 + 재무 장부 '미결'(출고번호 OUT-xxxx 생성)
2) [매출 등록]에서 '위탁 미결 내역 호출' → 미결 건 로드
3) ★부분 정산: 1,000 중 100만 확정, 900은 계속 미결
4) 확정 시: 재무 장부 매출 += 정산수량, 해당 출고번호 미결잔여 −= 정산수량
```
- **미결 원장** 상태: `[총 출고수량 / 기정산 / 미결잔여]`.
- **불변식**: `총 출고수량 = 기정산 + 미결잔여` (항상). **초과정산 방지** 검증 필수. **원본 출고번호 역추적**.
- 매출 확정 + 미결 차감은 **한 트랜잭션**(동시성 포함 정합성).

### 5.3 [로직 C] 주문 상태머신 + 상태 반영 (난이도: 중) — v0.3

```
[백오피스 내부]  신청완료 → (작업요청서 등록=물류 지시) → 출력대기 → 포장중 → 발송완료
```

- **진행상태 갱신 주체(v0.2 확정)**: DS물류 시스템 연동 없음(발주처 요청). 진행상태는 **담당자 수동 갱신**, DS물류로부터는 **수신 알람만**. ⚠️ **수동 갱신이므로 "누가·언제 치는가"를 운영 절차로 확정해야 함** — 담당자가 안 치면 특약점 화면이 안 바뀜(🔴 발주처 확인).

- **⚠️ 상태 어휘가 4개 문서에서 전부 다름 — 상태 전이 매트릭스가 별도 산출물로 필요:**

  잠정 매핑(승인 필요 🔴): `출력대기·포장중 → 프론트 "상품준비중"(수정잠금)` / `발송완료 → 프론트 "발송완료"`. **프론트 DB의 실제 enum 값을 소스에서 추출해 확정할 것.**

## 6. 외부·시스템 연동 명세

### 6.1 DSRE — 정체 ✅확정 / 연동방식 🔴미확정
- **정체 ✅**: **더프리미엄모의고사 운영 웹시스템**(0.1 근거). **더프모 주문만** order → DSRE로 유입(0.2). 교재·기타는 무관.
- **⚠️ 소스·스키마 확보됨** — 연동 방식 후보(API/DB직결/배치)는 문의가 아니라 **소스에서 직접 확인** 가능.
- **연동**: 매출일괄등록이 **DSRE 데이터를 read**. 현재 수기 → **자동 연동 필요**(100 vs 50 괴리 제거).
- **통합 검토**: DSRE를 매출프로그램에 통합해도 되는지 발주처에 제안 예정.
- 🔴 **확인 필요**: 연동 방식(API/DB/파일), 데이터 스키마, 테스트 계정·소스·DB.

### 6.2 order.dsdo.co.kr (특약점 간편신청 — 별도 트랙)
- **P.1~2 = 신규 목업 / P.3 = 기존 화면+버그**(0.5). 소스 확보됨 → 기존 [모의고사] 모듈 **복제·변형으로 신규 4화면 제작** 가능.
- **신규 "과목별 모의고사/교재" 페이지**(신청/조회·수정/거래명세서(물류연동)/재고현황)는 **매출프로그램 직결**(0.2 — DSRE 무관).
- ⚠️ **P.3 버그(상품준비 상태 수량 수정 가능)**: 서버 검증 누락이면 API 직접 호출로 뚫림 → 보안 이슈. 소스에서 원인 특정.
- 개선 요청: **상품 다중선택·수량 입력**(현재 불가), **담당자정보 자동 불러오기**, **전년도 정보 불러오기**(연 1회 리셋), 학년별 화면 분리 등(8장).
- 거래명세서 = 매출프로그램 **작업요청서** 화면과 연동.

### 6.3 홈택스(국세청)-?
- 계산서신고 = **표준 양식 엑셀 생성 → 사람이 수동 업로드**. 실시간 연동 아님(난이도 하).

### 6.4 물류처 / DS물류
- 리브커넥스·학력개발 본사로 **거래명세서 출력(문서)** 전달.
- ⚠️ **DS물류는 크게 안 쓰는 기능 → 시스템 연동 없음.** **수신 알람 수준만**(신규 출고/작업 발생 시 알림 표시) 구현. 진행상태는 담당자 수동 갱신(5.3).
- 🟡 리브커넥스는 분리된 계열사지만 프로그램 공용. **계열사 통합 적용 여부·공수는 추후 별도 미팅**.

### 6.5 더존 ERP
- 세무는 홈택스 직행이라 실시간 연동 거의 없음. **회계결산용 월말 파일 연동 잔여 여부만 재무팀 확인**(있어도 파일 export 수준).

## 8. DSRE / 특약점 사이트 수정 요청 (별도 트랙)

> 재구축(매출프로그램)과 분리해 관리. 담당: 더프리미엄 박혜영 과장(브리핑), 회신 1차 박희원 과장.

**order.dsdo.co.kr 개선**
- 상품 **다중선택 + 수량 입력** 가능(현재 불가).
- 지사 주문 시 **담당자정보 자동 불러오기** → DSRE 접수완료 내역에 표시.
- 학년별·상품별·담당자정보 **연 1회 리셋** → **전년도 정보 불러오기**(DSA와 동일 방식).
- 화면 **학년별 분리** 요청.

**DSRE(더프모) 요구 8건**
1. 관련 PPT 별도 수령·확인.
2. 실시일자: 상품시행관리 날짜 자동 + **수동 변경 가능**(STEP03 확인사항 실시일자 동일).
3. 성적표 부수 개선: 기본 1부, 2·3부 요청 케이스. 학원관리>성적표부수 기본값 0, **학교관리엔 항목 없음 → 신규 생성**. 성적처리 부서 자동 확인.
4. 성적처리 부서에서 데이터 그대로 가져가 부수 확인.
5. **응시인원 10명 이하 성적표 미제공** 규칙 → 기간 설정 시 10명 이상이면 조회 가능하게.
6. **월별 조회 로직 명확화**(예: 7월 부수를 5월로 잘못 조회하는 케이스 방지).
7. **이메일/문자 예약발송** 기능 신규.
8. (order 개선·전년도 불러오기 등 위와 연계).

## 10. 난이도 지도 & 개발 순서

### 10.1 난이도 3층
- **1층(상, 백엔드 심장)**: 재고 엔진(입고/이고/BOM·위탁 자동이고), 위탁 미결 정산(부분정산 누적). → **외부에 종속되지 않는 자체 도메인**.
- **2층(중~하, 대부분 재현)**: 매출 집계·응시현황 피벗·순매출(매입축)·채권/여신·수금·수불부·홈택스 엑셀.
- **3층(중, 신규 시각화)**: 대시보드 2종(목표 입력)·물류작업비·작업요청서(문서).

### 10.2 권장 개발 순서(의존성)
1. **기초 마스터 + RBAC**(거래처/창고/도서+BOM/학교/단가, 사용자·역할·권한).
2. **재고 트랜잭션 코어(로직 A)** + 제품수불부 검산.
3. **주문/출고 + 상태머신(로직 C)**(통합등록·수기팝업·작업요청서·작업결과).
4. **위탁 미결 정산(로직 B)**(매출 등록).
5. **매출 집계·조회**(통합매출·매출액정리·순매출·매출일괄등록[DSRE]).
6. **마감/채권**(수금→외상매출장→현황[담보], 수익신고·계산서신고[홈택스]).
7. **물류비·대시보드**(병렬 가능).
8. **연동**: DSRE read 자동화, 특약점 사이트/상태 반영(방식 확정 후).

### ✅ 정리됨
- 재고·물류 원장 = **매출프로그램 자체 관리**(이중장부·위탁창고·BOM 고유 로직).
- 재무/세무 종착지 = **홈택스 엑셀**(더존 아님). 매출 전표 = 자체 + 홈택스 파일.
- 더존 실시간 write(재고로직 재현) 최악 시나리오 **거의 배제**.
- **(B) 동일 재현 확정**(레거시 공존 없음).
- **주문 유입 경로 2갈래 확정**(0.2): 더프모=DSRE 경유 / 교재·기타=매출프로그램 직결.
- **P.1~2 = 신규 목업 / P.3 = 기존 화면+버그**(0.5). 프론트 소스 확보 → 기존 모듈 복제·변형 가능.
- **DSRE = 더프리미엄모의고사 운영 웹시스템**(0.1). "물류회사" 안내는 오정보.
