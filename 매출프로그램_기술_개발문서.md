## 1. 시스템 개요

### 1.1 도메인 한 문장
> **"위탁(선출고 후정산) 비즈니스를 지탱하는 재고·매출 상태머신."**
> 대성 학력개발은 교재·모의고사(시험지)를 제작해 특약점·학원·학교에 공급한다. 매출프로그램은 주문 접수 → 재고 → 출고 → 매출 확정 → 수금 → 세무신고 → 물류작업 전 과정을 관리하는 백오피스다.

### 1.2 판매 방식 2종(복잡도의 근원)
- **정상 판매**: 출고 즉시 매출 확정.
- **위탁 판매(선출고 후정산)**: 먼저 보내되 아직 매출 아님. 거래처가 정산해줄 때 매출 확정. → **재고 상태와 매출 상태를 각각 별도 추적**해야 함(이중 장부).

### 1.3 시스템 지형 (정정 반영)
| 시스템 | 성격 | 우리 관여 |
|---|---|---|
| 더존 출판 ERP | 상위 회계/결산 | 실시간 연동 거의 없음(있어도 파일 수준) — 재무팀 확인 |
| **매출프로그램(백오피스)** | 재구축 대상 | **본 문서 전부** |
| **DSRE** | 더프리미엄모의고사 처리 시스템(실질 처리) | 🔴 정체·연동방식 확정 후 데이터 연동 + 매출프로그램 통합 검토 |
| order.dsdo.co.kr(특약점 간편신청) | 특약점 주문 사이트(기존). PDF 목업은 희망안 | 🔴 신규 페이지·수정(별도 트랙) |

### 1.4 일정
- **개발 목표: 2026-10-31** / **실사용 개시: 2026-11-말**. 인력 집중 투입 필요(Phase 우선순위 결정 요인).

## 2. 시스템 구성 & 대상 범위

### 2.1 데이터 흐름 (정정된 실제 구조)
```
[특약점]
   └─ order.dsdo.co.kr (간편신청 사이트, 기존)   ← PDF Page1~3은 '희망 목업'(신규/수정 대상)
          │  주문
          ▼
       [ DSRE ] (더프리미엄모의고사 실질 처리 시스템)
          │  더프 출고/응시 데이터
          ▼  (매출일괄등록이 DSRE 데이터를 read → 현재는 수기라 괴리)
   ┌──────────────────────────────────────────────┐
   │            매출프로그램 (백오피스) — 재구축 대상       │
   │  주문/출고 · 매출 · 마감 · 물류/작업 · 기초관리       │
   │  자체 재고 엔진 + 이중 장부 + 위탁 미결 정산 + 채권/여신 │
   └──────────────────────────────────────────────┘
          │                         │
          ▼(엑셀)                    ▼(문서/데이터)
      홈택스(세금계산서)        물류처(리브커넥스/본사) · DS물류(수신 알람만, 연동X)
          ·                         ·
      더존(회계결산 잔여 여부 — 재무팀 확인, 있어도 파일 수준)
```

### 2.2 현재 핵심 니즈(미팅 확인)
- **DSRE ↔ 매출프로그램 매출 괴리 자동화**: 현재 DSRE는 100부인데 매출프로그램엔 실제 매출 50부만 수기 입력 → **DSRE에 안 들어가도 매출프로그램에서 매출수불 확인 가능**하도록 자동화하는 것이 핵심.
- **매출일괄등록**: DSRE에서 그대로 가져오는 방식.
- **매출액 정리**: 현재 수기 → 자동 표시 개선.
- **대시보드 신규**: 현재 RAW 엑셀 받아 수기 그래프 → 프로그램 내 대시보드로.

### 2.3 재구축 방침
- **(B) 동일 재현 확정**: 레거시 ASP 존치가 아니라, **기능/로직을 Java/React로 동일 재현**(레거시 공존 없음).
- 화면 유형: **[신규] 완전 신규 설계 / [개편] 기존+새 개념 반영 / [🔁재현] 기존 로직 이식**. 재현 화면은 **기존 ASP 소스·DB 확보가 전제**.

## 4. 핵심 데이터 모델 (5대 축 + α)

> 재구축 설계의 뼈대. 이 축들이 서로 얽혀 복잡도가 발생한다. (컬럼은 예시, 실제 스키마는 기존 DB 확보 후 확정)

### 4.1 상품(Product)
- **단품 ↔ 세트(BOM)**: 재고는 단품(회차별 낱권) 단위. 세트 재고는 단품을 **조립(BOM)** 해야 생성.
- **자체 교재 ↔ 외부 콘텐츠**: 자체는 매입원가 없음. 외부(이감·한석원 등)는 **매입 발생** → 순매출 계산식 다름.
```sql
products ( id, code, name, content_type,  -- SELF(자체) | EXTERNAL(외부매입)
           is_set BOOLEAN, ... )
bom_items ( parent_product_id, child_product_id, ratio )  -- 완제품→구성품+비율
```

### 4.2 창고(Warehouse) / 재고(Inventory)
- **물류창고(실물) ↔ 위탁창고(가상)**. 재고는 **상품 × 창고** 단위 원장.
```sql
warehouses ( id, name, type )   -- PHYSICAL(물류) | CONSIGN(위탁/가상)
inventory ( product_id, warehouse_id, qty )
inventory_txn ( id, product_id, warehouse_id, txn_type, qty, ref_no, created_at )
-- txn_type: INBOUND | OUTBOUND | TRANSFER | BOM_ASSEMBLE | BOM_DISASSEMBLE | DISPOSE | RETURN
```

### 4.3 장부(Ledger) — ★이중 장부(가장 중요)
- **물류(재고) 장부**: 물건 위치·수량. 나가면 즉시 차감.
- **재무(매출) 장부**: 매출 금액. 위탁은 나가도 즉시 매출 아님(미결).
- **핵심 규칙**: 위탁출고 → 물류 장부 즉시 차감 / 재무 장부 **미결(보류)** → 나중 정산 시 재무로 전환.

### 4.4 매출유형 / 회계구분 (룩업 자동매핑)
| 출고유형(6) | 회계구분(3) | 재고위치 |
|---|---|---|
| 정상출고 | 매출 | 물류창고(실물) |
| 위탁출고 | 매출(미결→확정) | **위탁창고(가상)** |
| 증정용 | 무상 | 물류창고 |
| 교사용 | 무상 | 물류창고 |
| 반품 | 반품 | 물류창고(음수) |
| 취소 | 반품 | — |
```sql
sales_txn ( id, product_id, partner_id, sales_type,  -- NORMAL(일반) | CONSIGN(위탁)
            out_type, acct_type, qty, supply_amount, tax, total,
            source_out_no,  -- 위탁 확정 시 원본 출고번호 역추적
            ... )
```

### 4.5 매입(Purchase) — 외부 콘텐츠
```sql
purchases ( id, product_id, partner_id, qty, purchase_return_qty, amount, ... )
-- 순매출: 매출(A) − 매입(B) = 이익, 이익률
```

### 4.6 채권(AR)/여신·담보
```sql
ar_ledger ( partner_id, opening_balance, ... )  -- 미수금 = 매출 − 수금 (러닝밸런스)
receipts ( id, partner_id, receipt_type, pay_type, amount, note_no, due_date, bank, ... ) -- 수금/어음
collaterals ( partner_id, amount, rate, expires_at, kind, content )  -- 담보/여신
```

### 4.7 목표(예산) — 대시보드용(실적과 독립)
```sql
sales_targets ( year, month, scope, scope_ref_id, target_amount )  -- 전사/제품별/거래처별
```

### 4.8 권한(RBAC) — 횡단
- 같은 화면·기능도 **부서별(사업부/재무팀/물류팀)** 접근 상이. 예: 입고/대체 화면에서 물류팀은 "일반입고"만.

### 4.9 🆕 매출 월마감(컨펌/잠금)
- 한 달 매출 처리를 모두 끝낸 뒤 **[월마감 컨펌] 버튼**을 누르면 해당 월 매출 데이터가 **수정 잠금**된다(더 이상 등록/수정/취소 불가).
```sql
period_locks (
  id, year, month, branch_id NULL,   -- (지점/계열사 구분 필요 시)
  status,        -- OPEN | CONFIRMED(잠금)
  confirmed_by, confirmed_at,
  reopened_by, reopened_at            -- 잠금 해제(재오픈) 이력 — 권한자만
)
```
- 잠금 판정: 매출/수금/전표 등 **쓰기 API는 대상 연월의 `period_locks.status`를 먼저 검사** → `CONFIRMED`면 거부(에러 `PERIOD_LOCKED`).
- **재오픈(잠금 해제)** 은 상위 권한(예: 재무팀/관리자)만 가능하며 이력 남김.

## 5. 3대 핵심 로직 (프로젝트의 심장)

> 난이도의 대부분이 여기 집중. 화면보다 **백엔드 도메인 정합성**이 핵심.

### 5.1 [로직 A] 재고 엔진 — 입고 / 이고 / BOM (난이도: 상)
모든 작업은 **원자적 트랜잭션**(중간 실패 시 전체 롤백) + 음수재고 방지.
- **(1) 일반 입고**: 인쇄소(예: 대성인쇄) → 물류창고 최초 입고. 단품 단위, **입고단가(원가) 기록**. `물류창고.단품 += 입고수량`.
- **(2) 단순 이고**: 매출 없이 위치 이동(물류창고→위탁창고 등). `출발 −=, 도착 +=`. **★위탁출고 선택 시 이 이고가 자동 백그라운드 생성**(주문 화면과 한 트랜잭션).
- **(3) BOM 조립/해체**: 세트 조립 100개 시 — 완제품 +100, 구성품 각 −(BOM비율×100). 해체는 부호 반대. **조립/해체에 물류작업비(포장비 등) 발생** → 물류비 연동.
```
[완제품] 국어세트 +100
 └ 국어1회차 −100 · 국어2회차 −100 · 국어3회차 −100   (BOM비율 1:1:1)
```
- **(4) 검산 = 제품수불부**: `이월 + 입고 − 매출 − 교사용 − 반품 ± 폐기 ± 대체 = 현재재고`. 이 표가 재고 엔진의 **인수 테스트 명세**.

### 5.2 [로직 B] 위탁 미결 정산 — 부분정산 누적 (난이도: 상)
```
1) 위탁출고 → 물류 장부 차감 + 재무 장부 '미결'(출고번호 OUT-xxxx 생성)
2) [매출 등록]에서 '위탁 미결 내역 호출' → 미결 건 로드
3) ★부분 정산: 1,000 중 100만 확정, 900은 계속 미결
4) 확정 시: 재무 장부 매출 += 정산수량, 해당 출고번호 미결잔여 −= 정산수량
```
- **미결 원장** 상태: `[총 출고수량 / 기정산 / 미결잔여]`.
- **불변식**: `총 출고수량 = 기정산 + 미결잔여` (항상). **초과정산 방지** 검증 필수. **원본 출고번호 역추적**.
- 매출 확정 + 미결 차감은 **한 트랜잭션**(동시성 포함 정합성).

### 5.3 [로직 C] 주문 상태머신 + 상태 반영 (난이도: 중) — 정정 반영
```
신청완료 → (작업요청서 등록=물류 지시) → 출력대기 → 포장중 → 발송완료
```
- **진행상태 갱신 주체(정정 v0.2)**: **DS물류는 크게 안 쓰는 기능이라 시스템 연동은 하지 않음.** 진행상태(출력대기/포장중/발송완료)는 **담당자 수동 갱신**을 기본으로 하고, DS물류로부터는 **수신 알람(신규 출고/작업 알림) 정도만** 표시한다. (양방향 pull 연동 아님)
- **프론트(order.dsdo.co.kr) 반영**: PDF 목업의 "상품준비중=수정잠금 / 신청완료=수정가능"은 **신규로 설계할 희망 동작**이며, 데이터는 프론트가 매출프로그램에 직접 붙는 게 아니라 **DSRE 경유** 가능성 → 동기화 방식(실시간/배치)·경로 확정 필요.

## 6. 외부·시스템 연동 명세

### 6.1 🔴 DSRE (최우선 규명)
- **정체**: 더프리미엄모의고사 처리 시스템(모의고사 출고/응시 데이터). order.dsdo.co.kr 주문이 DSRE로 유입.
- **연동**: 매출일괄등록이 **DSRE 데이터를 read**. 현재 수기 → **자동 연동 필요**(100 vs 50 괴리 제거).
- **통합 검토**: DSRE를 매출프로그램에 통합해도 되는지 발주처에 제안 예정.
- 🔴 **확인 필요**: 연동 방식(API/DB/파일), 데이터 스키마, 테스트 계정·소스·DB.

### 6.2 🔴 order.dsdo.co.kr (특약점 간편신청 — 별도 트랙)

## 8. DSRE / 특약점 사이트 수정 요청 (별도 트랙)

> 재구축(매출프로그램)과 분리해 관리. 담당: 더프리미엄 박혜영 과장(브리핑), 회신 1차 박희원 과장.

**order.dsdo.co.kr 개선**
- 상품 **다중선택 + 수량 입력** 가능(현재 불가).
- 지사 주문 시 **담당자정보 자동 불러오기** → DSRE 접수완료 내역에 표시.
- 학년별·상품별·담당자정보 **연 1회 리셋** → **전년도 정보 불러오기**(DSA와 동일 방식).
- 화면 **학년별 분리** 요청.

**DSRE(더프모) 요구 8건**
1. 관련 PPT 별도 수령·확인.
2. 실시일자: 상품시행관리 날짜 자동 + **수동 변경 가능**(STEP03 확인사항 실시일자 동일).
3. 성적표 부수 개선: 기본 1부, 2·3부 요청 케이스. 학원관리>성적표부수 기본값 0, **학교관리엔 항목 없음 → 신규 생성**. 성적처리 부서 자동 확인.
4. 성적처리 부서에서 데이터 그대로 가져가 부수 확인.
5. **응시인원 10명 이하 성적표 미제공** 규칙 → 기간 설정 시 10명 이상이면 조회 가능하게.
6. **월별 조회 로직 명확화**(예: 7월 부수를 5월로 잘못 조회하는 케이스 방지).
7. **이메일/문자 예약발송** 기능 신규.
8. (order 개선·전년도 불러오기 등 위와 연계).

## 10. 난이도 지도 & 개발 순서

### 10.1 난이도 3층
- **1층(상, 백엔드 심장)**: 재고 엔진(입고/이고/BOM·위탁 자동이고), 위탁 미결 정산(부분정산 누적). → **외부에 종속되지 않는 자체 도메인**.
- **2층(중~하, 대부분 재현)**: 매출 집계·응시현황 피벗·순매출(매입축)·채권/여신·수금·수불부·홈택스 엑셀.
- **3층(중, 신규 시각화)**: 대시보드 2종(목표 입력)·물류작업비·작업요청서(문서).

### 10.2 권장 개발 순서(의존성)
1. **기초 마스터 + RBAC**(거래처/창고/도서+BOM/학교/단가, 사용자·역할·권한).
2. **재고 트랜잭션 코어(로직 A)** + 제품수불부 검산.
3. **주문/출고 + 상태머신(로직 C)**(통합등록·수기팝업·작업요청서·작업결과).
4. **위탁 미결 정산(로직 B)**(매출 등록).
5. **매출 집계·조회**(통합매출·매출액정리·순매출·매출일괄등록[DSRE]).
6. **마감/채권**(수금→외상매출장→현황[담보], 수익신고·계산서신고[홈택스]).
7. **물류비·대시보드**(병렬 가능).
8. **연동**: DSRE read 자동화, 특약점 사이트/상태 반영(방식 확정 후).

| 20 | 목표 입력 세분화 단위(전사/제품별/거래처별), 집계 스냅샷 주기 |
| 21 | 리브커넥스 등 **계열사 통합 여부/공수**(별도 미팅) |
| 22 | 🆕 **매출 월마감(컨펌) 정책**: 잠금 단위(전사/지점/계열사), 재오픈 권한, **위탁 미결 잔여의 잠금 후 이월 처리** |

### ✅ 정리됨
- 재고·물류 원장 = **매출프로그램 자체 관리**(이중장부·위탁창고·BOM 고유 로직).
- 재무/세무 종착지 = **홈택스 엑셀**(더존 아님). 매출 전표 = 자체 + 홈택스 파일.
- 더존 실시간 write(재고로직 재현) 최악 시나리오 **거의 배제**.
- **(B) 동일 재현 확정**(레거시 공존 없음).
- **프론트는 기존 운영 화면이 아니라 희망 목업**, 실제 경로는 **DSRE 경유**(0장 정정).
